## 核心业务完善分析报告

### 严重级别问题（需立即修复）

#### 1. 定时任务未恢复库存 -- 数据一致性缺陷

`ScheduledTaskService.java:42-64` 自动取消超时订单时**只更新了状态，没有恢复商品库存**，导致库存永久丢失：

```java
// 当前代码：只改状态，漏掉库存恢复！
for (Order order : expiredOrders) {
    order.setOrderStatus("cancelled");
    orderRepository.save(order);
    // ❌ 缺少：恢复订单商品的库存
}
```

而 `OrderService.cancelOrder()` 手动取消时是有库存恢复的（第244-251行）。两者行为不一致。

---

#### 2. 骑手抢单存在并发竞态条件

`RiderService.java:140-179` 的 `grabOrder()` 方法**没有加任何锁机制**。多个骑手同时查询到同一可接订单并同时调用抢单接口时，可能出现**两个骑手都抢到同一单**的情况。

建议方案：在 Order 实体添加 `@Version` 乐观锁字段，或使用 `SELECT ... FOR UPDATE` 悲观锁。

---

#### 3. 购物车无库存上限校验

`CartService.java:50-76` 的 `addToCart()` 方法**没有校验购物车中商品数量是否超过实际库存**。用户可以将超出库存的商品加入购物车，下单时才报错，体验差且容易产生脏数据。

---

### 中等级别问题（建议尽快修复）

#### 4. Admin 统计看板存在性能隐患

`AdminService.java:276-278` 的 `getDashboardStats()` 使用了 `orderRepository.findAll()` **全表加载所有订单到内存做聚合统计**：

```java
List<Order> allOrders = orderRepository.findAll(); // ⚠️ 订单量大时 OOM
Map<String, Long> distribution = allOrders.stream()...
```

应改为数据库层面聚合（JPQL/Criteria 或原生 SQL GROUP BY）。

---

#### 5. 缺少订单异常状态处理

当前订单状态机只有正常流转路径，**缺少以下异常分支**：
| 场景 | 当前状态 | 建议补充 |
|------|---------|---------|
| 商家拒绝接单 | 无 | `pending → rejected` (含退款) |
| 骑手无法送达/异常 | 无 | `delivering → exception` |
| 用户拒收 | 无 | `delivered → rejected` |
| 配送超时无处理 | 无 | 定时任务检测超时订单 |

---

#### 6. 骑手配送流程缺少超时监控

- 骑手接单后多久必须取餐？**无约束**
- 取餐后多久必须送达？**无约束**
- 建议在 `ScheduledTaskService` 中增加：
  - 接单后 15 分钟未取餐 → 自动取消接单，释放回订单池
  - 配送超 60 分钟 → 标记异常，通知管理员

---

#### 7. 评价系统缺少业务约束

需要检查以下规则是否已实现（从代码推测可能缺失）：
- **是否限制只有 completed 状态的订单才能评价？**
- **是否防止同一订单重复评价？**
- **商家/骑手回复评价功能是否存在？**

---

### 低优先级改进（建议迭代完善）

#### 8. 支付金额二次校验

`PaymentService.createPayment()` 直接使用 `order.getTotalAmount()` 创建支付记录，虽然内部调用相对安全，但建议增加**请求参数金额与服务端计算金额的比对校验**，防止篡改。

#### 9. 通知/消息推送体系完全缺失

外卖平台的核心体验依赖实时通知，但当前系统中**没有任何通知机制**：
- 商家有新订单 → 无推送
- 骑手有可接单 → 无推送  
- 订单状态变更 → 用户无感知
- 建议引入 WebSocket 或消息队列（RabbitMQ/Kafka）

#### 10. 优惠券/满减活动模块空白

作为外卖平台，**营销体系完全缺失**：
- 无优惠券功能
- 无满减活动
- 无新用户首单优惠
- 这直接影响平台转化率和用户留存
